NYC Taxi Analytics Pipeline · Productionizing

Production on Snowflake, deployed by CI/CD

An exploratory project built at production level — the goal is learning the real warehouse + CI/CD workflow. Snowflake is the warehouse; GitHub Actions deploys the pipeline; everything is parameterized so it ports to any Snowflake account by swapping secrets.

The pipeline runs entirely on Snowflake: ingestion/ lands raw data and dbt/ builds the modeled layers. Ad-hoc analysis lives in analysis/ as Snowflake SQL.

Why Snowflake here

Learning the real stack

Stages + COPY, virtual warehouses, roles, and CI-driven dbt — the actual patterns you'd run in industry, on real cloud compute + storage.

Free to start, portable later

Begin on a 30-day trial ($400 credits — ample for 4 GB). Because the pipeline is account-agnostic, moving to a paid or different account is a secrets swap, not a rewrite.

Caveat to plan around: the free trial expires at 30 days / $400. The CI/CD design below makes re-pointing to a new Snowflake account trivial — so trial expiry never means a rebuild.

Connection is parameterized — the key to portability

No account identifier, user, or warehouse name is ever hard-coded. They live in GitHub Actions secrets. Point the whole pipeline at a different Snowflake account by editing secrets in the repo settings — zero code change.

# GitHub repo → Settings → Secrets and variables → Actions
SNOWFLAKE_ACCOUNT     # e.g. ab12345.eu-west-1
SNOWFLAKE_USER
SNOWFLAKE_PASSWORD    # or key-pair / OAuth for prod
SNOWFLAKE_ROLE        # e.g. SYSADMIN
SNOWFLAKE_WAREHOUSE   # e.g. COMPUTE_WH
SNOWFLAKE_DATABASE
SNOWFLAKE_SCHEMA

The CI/CD workflow

# .github/workflows/deploy.yml
on:
  push: { branches: [main], paths: ['dbt/**'] }   # deploy on model change
  schedule: [{ cron: "0 6 5 * *" }]            # monthly refresh after TLC publishes
  workflow_dispatch:                            # manual / backfill
jobs:
  deploy:
    runs-on: ubuntu-latest
    env:                                          # secrets → env, account-agnostic
      SNOWFLAKE_ACCOUNT: ${{ secrets.SNOWFLAKE_ACCOUNT }}
      SNOWFLAKE_USER: ${{ secrets.SNOWFLAKE_USER }}
      SNOWFLAKE_PASSWORD: ${{ secrets.SNOWFLAKE_PASSWORD }}
      SNOWFLAKE_WAREHOUSE: ${{ secrets.SNOWFLAKE_WAREHOUSE }}
      SNOWFLAKE_DATABASE: ${{ secrets.SNOWFLAKE_DATABASE }}
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: python ingestion/load.py          # land RAW
      - run: cd dbt && dbt seed && dbt run && dbt test   # model + test
A new Snowflake account? Update the secrets, re-run the workflow. The ingestion/ + dbt/ code never changes — that's what "portable" means in practice.

Ingest → model: how data actually lands

The public s3://nyc-tlc bucket denies anonymous access, so ingestion pulls Parquet from the CloudFront mirror and COPYs it into a RAW table. dbt then builds the modeled layers.

-- ingestion/sql — land RAW (PUT from CloudFront download, then COPY)
CREATE TABLE IF NOT EXISTS raw_yellow_tripdata (
  tpep_pickup_datetime  TIMESTAMP_NTZ, tpep_dropoff_datetime TIMESTAMP_NTZ,
  passenger_count NUMBER, trip_distance FLOAT,
  PULocationID NUMBER, DOLocationID NUMBER, payment_type NUMBER,
  fare_amount FLOAT, tip_amount FLOAT, total_amount FLOAT,
  _loaded_at TIMESTAMP_NTZ DEFAULT CURRENT_TIMESTAMP()   -- drives source freshness
);

COPY INTO raw_yellow_tripdata FROM @raw_stage
  FILE_FORMAT=(TYPE=parquet)
  MATCH_BY_COLUMN_NAME=CASE_INSENSITIVE;   -- skips already-loaded files

Production hardening checklist

Connection fully parameterized via secrets — portable to any account
Models versioned in dbt/; idempotent rebuilds + incremental fct_trips
DQ filters live once in the dbt intermediate layer — single source of truth
Deploy on push to dbt/** + scheduled refresh (Airflow DAG) + manual dispatch
Key-pair / OAuth auth for prod (password only for trial exploration)
Least-privilege role; warehouse auto-suspend to cap credit burn
