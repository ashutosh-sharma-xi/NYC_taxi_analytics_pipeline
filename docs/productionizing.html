NYC Taxi Analytics Pipeline · Productionizing

Production on Snowflake, deployed by CI/CD

An exploratory project built at production level — the goal is learning the real warehouse + CI/CD workflow. Snowflake is the warehouse; GitHub Actions deploys the pipeline; everything is parameterized so it ports to any Snowflake account by swapping secrets.

DuckDB still powers the local notebooks/ for fast, free exploration. Production runs on Snowflake — the SQL written in the notebooks is warehouse-portable, so it lifts over unchanged.

Why Snowflake here

Learning the real stack

External tables, virtual warehouses, roles, and CI-driven DDL — the actual patterns you'd run in industry, on real cloud compute + storage.

Free to start, portable later

Begin on a 30-day trial ($400 credits — ample for 4 GB). Because the pipeline is account-agnostic, moving to a paid or different account is a secrets swap, not a rewrite.

Caveat to plan around: the free trial expires at 30 days / $400. The CI/CD design below makes re-pointing to a new Snowflake account trivial — so trial expiry never means a rebuild.

The public TLC Parquet is read by a Snowflake external table — no ingestion copy needed. CI applies the DDL idempotently; a monthly schedule refreshes for new data.

Connection is parameterized — the key to portability

No account identifier, user, or warehouse name is ever hard-coded. They live in GitHub Actions secrets. Point the whole pipeline at a different Snowflake account by editing secrets in the repo settings — zero code change.

# GitHub repo → Settings → Secrets and variables → Actions
SNOWFLAKE_ACCOUNT     # e.g. ab12345.eu-west-1
SNOWFLAKE_USER
SNOWFLAKE_PASSWORD    # or key-pair / OAuth for prod
SNOWFLAKE_ROLE        # e.g. SYSADMIN
SNOWFLAKE_WAREHOUSE   # e.g. COMPUTE_WH
SNOWFLAKE_DATABASE
SNOWFLAKE_SCHEMA

The CI/CD workflow

# .github/workflows/deploy.yml
on:
  push: { branches: [main], paths: ['sql/**'] }   # deploy on SQL change
  schedule: [{ cron: "0 6 5 * *" }]            # monthly refresh after TLC publishes
  workflow_dispatch:                            # manual / backfill
jobs:
  deploy:
    runs-on: ubuntu-latest
    env:                                          # secrets → env, account-agnostic
      SNOWFLAKE_ACCOUNT: ${{ secrets.SNOWFLAKE_ACCOUNT }}
      SNOWFLAKE_USER: ${{ secrets.SNOWFLAKE_USER }}
      SNOWFLAKE_PASSWORD: ${{ secrets.SNOWFLAKE_PASSWORD }}
      SNOWFLAKE_WAREHOUSE: ${{ secrets.SNOWFLAKE_WAREHOUSE }}
      SNOWFLAKE_DATABASE: ${{ secrets.SNOWFLAKE_DATABASE }}
      SNOWFLAKE_SCHEMA: ${{ secrets.SNOWFLAKE_SCHEMA }}
    steps:
      - uses: actions/checkout@v4
      - run: pip install snowflake-connector-python
      - run: python deploy/run_sql.py sql/    # executes versioned .sql files in order
A new Snowflake account? Update the six secrets, re-run the workflow. The sql/ files never change — that's what "portable" means in practice.

The warehouse DDL — same model as the notebooks

External table over the public Parquet, then the same DQ filters + star schema proven in notebook 03.

-- sql/01_external_table.sql
CREATE STAGE IF NOT EXISTS tlc_stage
  URL='s3://nyc-tlc/trip data/'
  FILE_FORMAT=(TYPE=parquet);

CREATE EXTERNAL TABLE IF NOT EXISTS ext_yellow_2023 (
  tpep_pickup_datetime  TIMESTAMP AS (value:tpep_pickup_datetime::timestamp),
  tpep_dropoff_datetime TIMESTAMP AS (value:tpep_dropoff_datetime::timestamp),
  passenger_count       INT       AS (value:passenger_count::int),
  trip_distance         FLOAT     AS (value:trip_distance::float),
  PULocationID          INT       AS (value:PULocationID::int),
  DOLocationID          INT       AS (value:DOLocationID::int),
  fare_amount           FLOAT     AS (value:fare_amount::float),
  tip_amount            FLOAT     AS (value:tip_amount::float),
  total_amount          FLOAT     AS (value:total_amount::float),
  payment_type          INT       AS (value:payment_type::int)
)
  LOCATION=@tlc_stage
  PATTERN='.*yellow_tripdata_2023-.*[.]parquet'
  FILE_FORMAT=(TYPE=parquet);
-- sql/02_fact_trips.sql  — same DQ filters as notebook 02
CREATE OR REPLACE TABLE fact_trips AS
SELECT
  tpep_pickup_datetime::date          AS date_key,
  PULocationID                        AS pu_location_id,
  DOLocationID                        AS do_location_id,
  payment_type, trip_distance,
  fare_amount, tip_amount, total_amount
FROM ext_yellow_2023
WHERE fare_amount > 0 AND trip_distance > 0
  AND tpep_dropoff_datetime > tpep_pickup_datetime
  AND passenger_count BETWEEN 1 AND 6
  AND EXTRACT(YEAR FROM tpep_pickup_datetime) = 2023;

Production hardening checklist

Connection fully parameterized via secrets — portable to any account
SQL versioned in sql/, applied in order, idempotent (CREATE OR REPLACE)
DQ filters identical to notebook 02 — single source of truth for cleaning
Deploy on push to sql/** + scheduled monthly refresh + manual dispatch
Key-pair / OAuth auth for prod (password only for trial exploration)
Least-privilege role; warehouse auto-suspend to cap credit burn
