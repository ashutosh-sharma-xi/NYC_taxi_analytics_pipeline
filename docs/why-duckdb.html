Same SQL, two homes: prototype a query against DuckDB in a notebook, then promote the logic into a dbt model that runs on Snowflake. Portable SQL is what makes that handoff free.

Why not just use one for both?

DuckDB everywhere?

Great for a laptop, but no shared warehouse, roles, scheduling, or concurrent BI — the things a production pipeline needs.

Snowflake everywhere?

Needs an account + warehouse spin-up just to poke at the data. Slower, costlier loop for quick exploration.

Verdict

Use each engine where it's strongest: DuckDB for the fast, free, offline exploration loop in the notebooks, and Snowflake as the production warehouse the dbt project builds and tests. Warehouse-portable SQL lets the same logic move from one to the other with no rewrite.
