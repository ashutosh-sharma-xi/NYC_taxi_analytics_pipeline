Understanding DuckDB

A primer: what DuckDB is, how it compares to Snowflake, and when each is the right tool. This project runs on Snowflake — DuckDB is documented here for context, since it's the tool you'd reach for to explore a Parquet file in seconds.

What is DuckDB?

🦆
"SQLite for analytics"An in-process analytical (OLAP) database. It runs inside your Python/CLI process — no account, no server, no cluster.
📦
Reads files directlyQueries local or remote Parquet/CSV in place — no load step, no staging. Only the columns/rows a query needs are fetched.
⚡
Genuinely fast on one machineColumnar + vectorised execution makes it quick for GB-scale aggregations on a laptop.
🔌
Portable SQLANSI-style SQL that lifts cleanly to Snowflake / BigQuery — prototype locally, promote to the warehouse.

When to use which

Reach for DuckDB when…

Exploring/prototyping a query, single-analyst work on data that fits one machine, embedding analytics in an app/CI/notebook, or transforming Parquet/CSV with no infra.

Reach for Snowflake when…

Many concurrent users / BI dashboards, a shared governed always-on warehouse with roles, data beyond one machine, or a managed production pipeline (this project).

The one-liner

DuckDB is the fast, free exploration engine; Snowflake is the production warehouse. They share portable SQL, so logic prototyped in DuckDB moves to Snowflake with no rewrite. For a managed, multi-user, always-on pipeline like this one, Snowflake is the right home.
